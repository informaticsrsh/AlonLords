# Концепція: Фентезі-автобатлер з Лордами

## 0. Ігровий цикл (перша ітерація)
Перша ітерація — простий нескінченний луп без мета-прогресії між "забігами":
```
Підготовка (Hub, п.4) → Вибір шляху (3 варіанти, п.4) → Бій (п.5) → Підготовка → ...
```
- **Життя забігу**: у гравця є 3 життя. Програш бою (п.5, "Умова поразки") коштує 1 життя — гравець повертається в Hub і продовжує забіг (не game over). Втрата 3-го життя — game over, забіг завершується.
- **Складність**: зростає з кожним пройденим кроком циклу (кожен бій = +0.5 до лічильника складності). Програш бою суттєво відкидає складність назад — приблизно на 5 кроків прогресу (`difficulty -= 2.5`), а не скидає її повністю.

## 1. Лорд і характеристики

### Фракції лорда
```
LordType {
  id, name,
  primary_faction: faction_id,        // основна фракція — дає доступ до faction_bonuses
  secondary_factions: [faction_id],   // 0-1 додаткові фракції — лише доступ до найму юнітів цієї фракції, без бонусів
  faction_bonuses: [Modifier],        // прив'язані саме до primary_faction
}
```
- Найм юнітів обмежений `primary_faction` + `secondary_factions` лорда (базове правило; розширення — через спецнавик, як і раніше).
- Фракційний бонус (`faction_bonuses`) діє лише на юнітів `primary_faction` — юніти, найняті через `secondary_factions`, доступу до бонусу не мають.
- Лорд з 1 додатковою фракцією — гібрид (напр. Імперія як основна + Некрополь як додаткова): наймає юнітів обох, але бонус отримує тільки армія Імперії.

### Фракційний бонус (окремий тип Modifier з п.2)
```
FactionBonus {
  applies_to_faction: faction_id,
  trigger: "passive" | ON_BATTLE_START | ON_UNIT_KILL | ... ,
  condition: fn(unit) -> unit.faction == applies_to_faction,
  effect: fn(unit/state) -> state
}
```
Приклади:

**Імперія — "Бойовий дух" (стейтфул-механіка, не просто пасивний бафф):**
```
BattleSpiritState (per battle, belongs to lord's army):
  value: int, init = 50

Modifiers:
  ON_UNIT_KILL (killer.faction == "empire"):
    battleSpirit += faith_points(killed_unit)   // напр. exp_reward_on_kill вбитого як основа

  ON_UNIT_DEATH (dead.faction == "empire"):
    battleSpirit -= loss_penalty                // фікс/формула втрати духу

  ON_TURN_START (passive, applies_to: усі empire-юніти армії):
    if battleSpirit > 80: speed_modifier = 1.2   // окремий множник "швидкості дій", НЕ саме attack_speed
    if battleSpirit < 20: speed_modifier = 0.8
    else: speed_modifier = 1.0

  effective_attack_speed = unit.attack_speed / speed_modifier
  // speed_modifier > 1 → effective_attack_speed менший → юніт діє частіше (бонус)
  // speed_modifier < 1 → effective_attack_speed більший → юніт діє рідше (штраф)
```
Тобто "Бойовий дух" — армійський показник (не юніта і не лорда окремо), який зростає за вбитих ворогів і падає за втрачених своїх, а його поточний рівень через пороги (>80 / <20) масштабує окремий `speed_modifier`, який ділить базовий `attack_speed` кожного empire-юніта — сам `attack_speed` (тіки між діями) не змінюється, змінюється лише ефективне значення в бою.

**Термінологія**: "Бойовий дух" (`battleSpirit`) і "Віра" — одна й та сама механіка; "Віра" — лорний/UI-варіант назви для Імперії конкретно (напр. у навичках лорда Генріха), `battleSpirit` — технічне ім'я поля в коді. В інших документах ці терміни взаємозамінні.

- Лорд Орди Хаосу: "demon-юніти відновлюють 5% HP при вбивстві ворога"
- Лорд Некрополя: "undead-юніти не потребують воскресіння за золото — автопіднімаються з 50% HP після бою"

Бонус діє тільки на юнітів `primary_faction` лорда в армії. Юніти з `secondary_factions` в бою присутні, але під дію `faction_bonuses` не потрапляють — секондарна фракція дає лише доступ до найму, не до бонусу.

### Базові атрибути
| Атрибут | Ефект |
|---|---|
| Лідерство | Ліміт "ваги" армії: Σ(leadership_cost юніта) ≤ Leadership лорда |
| Тактика | Розмір сітки розміщення: старт 3×2, кожні N очок тактики → +1 клітинка (ряд/стовпець почергово), макс 6×4 |
| Бойова сила | Впливає на шкоду юніта через `Action.formula(lord, self)` (п.6), напр. `8 + 1*lord.battlePower` |
| Витривалість | Впливає на HP юніта через `UnitType.hp_formula(lord)` (п.6), напр. `60 + 1*lord.vitality` |
| Обсяг кристала | Стартовий запас мани лорда: Mana_max = h(CrystalVolume) |
| Швидкість відновлення | Реген мани: Mana_regen = k(RegenSpeed) — це значення "за 5 сек", застосовується по частинах щосекунди: `+Mana_regen / 5` кожен тік (тік = 1 реальна секунда) |

Немає окремих полів `attack_base`/`hp_base` — кожен юніт зберігає готову формулу (`hp_formula`, `Action.formula`), яка сама приймає характеристику лорда як аргумент і повертає фінальне значення. Це єдина модель нарахування статів у всьому документі (див. п.6); "base + f_unit(stat)" вище — лише спрощений запис того самого, а не окрема система.

### Розмір сітки від тактики
```
tactic_grid(tactic_points):
  base = (3,2)
  steps = tactic_points // N
  rows += steps // 2 (round-robin: chwyt ряд/стовпець по черзі)
  cols += steps - (steps // 2)
  clamp to (6,4)
```

## 2. Модульна система правил (щоб лорд міг "ламати" стандартні закони бою)

Архітектура: **Event Bus + Modifier Pipeline**, а не жорсткі if/else в коді бою.

### Ключова ідея
Кожна механіка бою (частота атаки, порядок ходів, дальність, шкода, зцілення тощо) — це **базове значення + список модифікаторів**, які застосовуються через events.

```
BattleEvent: {type, source, target, payload}
Types: ON_BATTLE_START, ON_TURN_START, ON_UNIT_ATTACK, 
       ON_UNIT_KILL, ON_UNIT_DEATH, ON_DAMAGE_DEALT, 
       ON_SKILL_CAST, ON_MANA_TICK ...
// Бій іде в реальному часі: "тік"/"хід" (ON_TURN_START, ON_MANA_TICK) = 1 реальна секунда,
// а не дискретна черга по юніту. attack_speed кожного юніта — це його власний кулдаун у тіках
// відносно цього спільного секундного таймера (див. Mana_regen у п.1 як приклад).

Modifier: {
  trigger: EventType,
  condition: fn(eventCtx) -> bool,
  effect: fn(state) -> state,  // напр. зменшити attack_cooldown
  priority: int,
  duration: "battle" | "stacks" | "permanent"
}
```

Приклад навички лорда з ТЗ ("вбив першого ворога → коротший кулдаун атаки"):
```
Modifier {
  trigger: ON_UNIT_KILL,
  condition: ctx => ctx.killCountThisBattle == 1,
  effect: state => state.units.forEach(u => u.attackCooldown *= 0.8),
  duration: "battle"
}
```

Всі "стандартні правила" (напр. "юніт атакує раз на T ходів") самі є базовим модифікатором з priority=0, тож навички лорда/юнітів (priority>0) можуть його перекривати, підмінювати чи скасовувати — без зміни ядра бойової системи.

### Практична вигода
- Нові лорди/юніти = нові дата-файли з модифікаторами, без правок ядра.
- Конфлікти вирішує priority + порядок реєстрації, дефолтна модель — "стакається", але конкретний модифікатор може задати replace/override.

## 3. Бойова економіка

### Нагороди за бій
Нагорода за бій — не тільки золото, а пул можливих типів (конкретний набір залежить від бою/сценарію):

```
BattleReward {
  gold: int,
  unique_units: [UnitType],       // рідкісні юніти, недоступні за звичайний найм
  economic_limit: int,            // приріст "економічного ліміту" (див. нижче)
  mine_capture_available: bool    // чи відкривається можливість захопити рудник після цього бою
}
```

**Економічний ліміт** — окремий ресурс (аналог "населення"):
- Пасивно генерує невеликий дохід золота щохід.
- Витрачається на: найм нових юнітів (додатково до вартості золотом) і на захоплення рудників.
- Тобто це "стеля" експансії, не сам дохід — щоб не можна було нескінченно наймати/захоплювати лише за золото.

**Рудник (Mine):**
```
Mine {
  economic_limit_cost: int,   // одноразова витрата ліміту на захоплення
  gold_income_per_turn: int   // великий пасивний дохід після захоплення
}
```
Захоплення рудника — разова дія, що постійно "з'їдає" частину economic_limit гравця в обмін на значний пасивний дохід золота щохід.

### Базова видача
- Кожен юніт, що вижив у бою, отримує **рівну частку** пулу досвіду бою:
```
exp_per_unit = battle_exp_pool / count(units_survived)
```
- Полеглі юніти досвід не отримують.

## 4. Після бою (Hub-фаза)

Дії лорда за золото:
1. **Найм** нових юнітів (обмежено лідерством).
2. **Лікування** живих поранених юнітів.
3. **Воскресіння** полеглих (дорожче за лікування).
4. **Підвищення (evolution)** — при досягненні exp-порогу юніт обирає одну з 2+ гілок:
   - Лучник → Снайпер (дальність/крит) або Арбалетчик (пробиття броні/шкода)
   - Мечник → Авангардист (танк/провокація) або Дуелянт (швидкість/одиночна ціль)
   - Перехід незворотній, суттєво змінює base-статистику і, можливо, набір навичок/модифікаторів юніта.

Структура даних апгрейду:
```
UnitClass {
  id, base_stats, skills[],
  evolutions: [
    { id, requirements: {level}, new_base_stats, new_skills[] }
  ]
}
```

### Вибір шляху (завершує Hub-фазу)
Коли гравець виконав усі дії підготовки, йому на вибір дається **3 шляхи**. Кожен шлях наперед визначає:
- набір ворогів наступного бою;
- бонус, який гравець отримає за проходження саме цього шляху (напр. `unique_units`, `economic_limit`, `mine_capture_available` — типи з `BattleReward`, п.3).

Гравець обирає один шлях — саме з його набором ворогів відбувається наступний бій; два інші варіанти для цього переходу пропадають.

## 5. Бій — автономна поведінка юнітів (Pre-battle Tactics UI)

### Умова поразки
Сторона програє бій, коли на полі не залишається жодного **живого юніта під її прямим контролем**. Прикликані істоти (summon) до цієї умови не рахуються: якщо у сторони лишились тільки прикликані істоти (жоден оригінальний найманий юніт не вижив), сторона вважається переможеною, а всі її прикликані істоти одразу зникають з поля.

### Рух у бою
Юніти не рухаються по сітці самостійно протягом бою — позиція, задана перед боєм (п.1, тактика/сітка), лишається фіксованою. Виняток: навички окремих юнітів можуть примусово перемістити ціль (свою чи ворожу) на іншу клітинку — це `Action` з відповідним `effect_kind` (напр. `"reposition"`), а не самостійний рух.

### Дальність і фронтлайн
- **Фронтлайн** — перший ряд ворожої сітки (найближчий до атакуючого), в якому є хоч один живий юніт. Якщо цей ряд повністю порожній, фронтлайном стає наступний ряд.
- **Ближній тип атаки (melee)** може цілити лише в межах фронтлайну: клітинку навпроти атакуючого (той самий стовпець) і сусідні стовпці цього ж ряду.
- **Дальній тип атаки (ranged)** ігнорує фронтлайн і може цілити будь-якого живого юніта на полі.
- `target_rule.selection` (nearest/lowest_hp/highest_threat/…) з п.6 шукає ціль **серед юнітів, доступних за цим правилом дальності**, а не по всій сітці одразу.
- Сітка ворога влаштована за тими самими правилами, але може мати інший розмір (більше/менше клітинок), ніж сітка гравця (п.1).

Перед боєм гравець задає для кожного юніта:
- **Пріоритет цілі**: найближча / найменше HP / найбільша загроза (dps) / певний тип (маг/танк) / останній атакований.
- **Пріоритет дії**: атака / навичка X / навичка Y (з ваговим чи послідовним порядком).
- **Тригери зміни поведінки** (умовні правила, IF-THEN):
```
Trigger {
  condition: HP% < 30 | ally_died | mana >= X | enemy_count < N | ...
  action: switch_priority(newPriorityList) | force_skill(id) | retreat | focus_target(rule)
}
```

Це той самий Modifier/Event механізм з п.2 — тригери гравця це просто user-defined модифікатори з умовою на основі event ON_TURN_START/ON_UNIT_DAMAGED тощо. Один рушій для правил лорда, навичок юнітів і гравцевої тактики.

### Позиційна цінність розміщення (авангард, сусіди, прикриття)
Розміщення на сітці — не просто дальність до цілі, а окремий шар балансу. Базові принципи:

- **Власний авангард** — перший ряд *своєї* сітки (найближчий до ворога), симетрично до "фронтлайну ворога" вище.
- **Сусіди клітинки** — клітинки, що межують по стороні (не по діагоналі) в тій самій сітці: той самий ряд ±1 стовпець, той самий стовпець ±1 ряд.
- Юніт може мати `positional_modifiers` (п.6) — правила, що змінюють його статистики залежно від того, в якому ряду він стоїть і хто розміщений на сусідніх клітинках. Це дозволяє:
  - **Штраф за авангард**: дальній юніт (лучник) втрачає частину шкоди/має шанс "збою" атаки, якщо стоїть у власному авангарді без прикриття.
  - **Захисник сусідів**: юніт типу "танк" перехоплює частину шкоди, спрямованої на сусідні клітинки (не лише себе) — `Guardian`-механіка (п.6).
  - **Каster б'є по сусідах цілі**: заклинання цілиться в 1 юніта, але шкода/ефект зачіпає і клітинки, сусідні саме до цілі (а не до самого каста), на відміну від класичного AoE навколо себе.
  - **Атака одразу двох клітинок**: `Action.target_rule.width` (п.6) — meleeудар б'є основну ціль + N сусідніх клітинок того ж ряду одночасно, незалежно від `all_in_range`/AoE-радіуса.

## 6. Характеристики юніта (data schema)

```
UnitType {
  id, name, class_tier,       // напр. "archer", tier 1

  // --- Приналежність ---
  faction: "empire" | "chaos_horde" | "undead" | "elves" | ... ,   // хто наймає юніта
  race_type: "human" | "demon" | "beast" | "undead" | "elf" | "construct" | ... ,
  // приклади пар faction+race_type:
  //   Імперія:      human
  //   Орда Хаосу:   demon, beast
  //   Некрополь:    undead
  // race_type може впливати на: специфічні resistances/vulnerabilities,
  // взаємодію з навичками (напр. некромант призиває тільки з трупів undead/human),
  // синергії лорда ("+X% атаки демонам")

  // --- Прогрес ---
  exp_current: int,
  exp_to_upgrade: int,        // ліміт для підвищення рангу
  exp_reward_on_kill: int,    // скільки дає ворогам за вбивство цього юніта (для оцінки складності бою)

  // --- Бойові дії ---
  attack_speed: float,        // "тіків" між діями; менше = частіше діє
  actions: [Action],          // звичайна атака теж Action, без спецвиключень

  // --- Здоров'я ---
  hp_formula: fn(lord) -> int,  // напр. 60 + 1 * lord.vitality

  // --- Захист ---
  resistances: {
    physical: int,   // фіксоване число, 0 за замовчуванням
    fire: int,
    holy: int,
    ...              // за кожним типом шкоди своя фіксована цифра, не залежить від лорда
  },

  // --- Позиція/розмір на сітці ---
  leadership_cost: int,
  grid_footprint: {rows, cols},
  positional_modifiers: [PositionalModifier],  // ефекти залежно від ряду/сусідів (див. нижче)

  evolutions: [...]  // див. п.4
}
```

### PositionalModifier (штрафи/бонуси за місце на сітці)
Той самий Modifier з п.2, але `condition` перевіряє не бойову подію, а поточну позицію юніта відносно власної сітки:
```
PositionalModifier {
  id, trigger: "passive",
  condition: "is_own_vanguard"                    // юніт стоїть у власному авангарді (перший ряд до ворога)
            | "is_own_vanguard_unshielded"         // авангард І немає melee-юніта на сусідній клітинці того ж ряду
            | "is_own_backline"                    // юніт стоїть у найдальшому від ворога ряду власної сітки
            | "no_adjacent_allies"                 // жодного союзника на сусідній клітинці
            | "adjacent_ally_type(race_type|class_tier)"
            | "adjacent_ally_melee"                 // є melee-союзник на сусідній клітинці того ж ряду ("упор" для важкої зброї)
            | "adjacent_ally_count(>=N)",           // кількість союзників на сусідніх клітинках не менша за N
  effect: fn(unit) -> unit   // напр. -25% до formula, +evasion, -resist тощо
}
```
Цей enum — базовий і відкритий: нові позиційні умови можна додавати за тим самим принципом (перевірка стану сітки навколо юніта), головне — щоб кожна нова умова, використана в даних юніта, спершу з'являлась тут.
Приклад (штраф лучника в авангарді, з ТЗ):
```
{ id:"exposed_in_front", condition:"is_own_vanguard_unshielded", effect: u => u.action_formula *= 0.7 }
```
Лучник, що випадково опинився в передньому ряду без прикриття танка поруч, атакує на 30% слабше — розміщення позаду стає відчутно вигіднішим, а не просто "правильним за лором".

### Action (будь-яка дія юніта, включно зі звичайною атакою)
```
Action {
  id, name,
  type: "physical" | "fire" | "holy" | "poison" | ... ,  // тип шкоди/ефекту, звіряється з resistances цілі
  effect_kind: "damage" | "heal" | "summon" | "buff" | "debuff" | "control" | "reposition" ...,   // reposition: примусово змінює клітинку цілі (свій/ворожий юніт не рухається сам, п.5)
  range_type: "melee" | "ranged",   // melee обмежений фронтлайном, ranged б'є будь-куди на полі (п.5)

  formula: fn(lord, self) -> value,
  // приклади:
  //   лучник:  base_attack -> 8 + 1 * lord.battlePower
  //   цілитель атака: 5 + 0.5 * lord.battlePower   (type: physical)
  //   цілитель зцілення: 10 + 1 * lord.crystalRegenSpeed  (type: holy)
  //   некромант summon: hp/attack нового зомбі = fn(lord, self)

  target_rule: {
    side: "enemy" | "ally" | "self",
    selection: "nearest" | "lowest_hp" | "highest_threat" | "random" | "all_in_range" | "corpse_of_dead_ally", 
    count: int | "aoe_radius" | "aoe_radius+N" | "all",  // "all" = б'є все поле (ультимативні здібності)
    width: int | null,                    // б'є основну ціль + (width-1) сусідні клітинки того ж ряду цілі, одночасно; незалежно від aoe_radius
    include_target_neighbors: bool,       // true = ефект зачіпає ще й клітинки, СУСІДНІ ДО ЦІЛІ (а не до кастера) — для каста по площі навколо обраної точки
    aoe_shape: "cross" | "diagonal" | "circle" | "column" | "row" | null,  // форма площі ураження, коли самого include_target_neighbors/width недостатньо (напр. відрізнити "хрест" від "кола"); null = звичайна ціль без форми
    aoe_origin: "self" | "target" | null   // від чого рахується aoe_shape: від кастера чи від обраної цілі
  },

  condition: fn(target) -> bool | null,   // додаткова умова застосування дії понад target_rule (напр. "лише якщо hp_pct цілі < X%"); null = завжди застосовна
  cooldown: int | null | "once_per_battle" | "once_per_N_battles",   // int = кулдаун у тіках; рядкові варіанти — для рідкісних "ультимативних" здібностей поза тіковою системою
  mana_cost: int | null,  // якщо дія лорда/юніта споживає кристал лорда

  // --- Додаткові модифікатори дії (опційні, залежать від типу юніта) ---
  crit_chance_bonus: float | null,            // + до базового шансу криту саме цієї дії
  ignore_resistance_pct: float | null,        // яку частку resistances цілі дія ігнорує (пробиття броні)
  on_hit_debuff: {id, effect, duration} | null,  // дебафф, що накладається на ціль при влучанні
  bonus_vs_race_type: {race_type: float, ...} | null,  // множник шкоди проти конкретних race_type (п.6)
  execute_on_crit_below_hp_pct: float | null, // при криті — миттєво добиває ціль, якщо її hp_pct нижче цього порогу
  bonus_if_target_non_frontline: float | null // множник шкоди, якщо ціль стоїть не на фронтлайні (п.5) свого боку
}
```
Усі поля розділу "Додаткові модифікатори дії" — опційні надбудови над базовою дією; юніт, якому вони не потрібні, просто лишає їх `null`. Це той самий підхід, що й з `cooldown`/`mana_cost` вище: базова дія працює і без них.

### Guardian (захист сусідніх юнітів, не тільки себе)
Окремий пасивний тип для "танків", що прикривають розміщені поруч клітинки:
```
Guardian {
  trigger: ON_TARGET_SELECTED,             // ворог обрав ціль для атаки/каста
  shape: "sides" | "behind" | "full",       // які саме сусідні клітинки прикриває: лише фланги (лівий/правий по ряду), лише позаду себе, чи всі 4 напрямки
  condition: ctx => ctx.target.side == guardian.side
                  && is_adjacent_by_shape(ctx.target, guardian, guardian.shape)
                  && guardian.alive,
  effect: ctx => redirect_pct(ctx, guardian, pct)   // pct шкоди йде по guardian замість оригінальної цілі
}
```
На відміну від ауро-баффу (`bastion_aura`, що просто зменшує вхідну шкоду сусідам), `Guardian` фізично перехоплює частину атаки на себе — цінність клітинки поруч із таким юнітом різко зростає. `shape` дозволяє різним "танкам" прикривати по-різному (напр. `full` дорожчий і рідкісний, `sides`/`behind` — дешевші й вужчі за покриттям), щоб однотипні Guardian-юніти не дублювали одне одного.

Приклади (з ТЗ):
- Лучник, звичайна атака: `type: physical, formula: 8 + 1*battlePower, target: enemy/nearest`
- Цілитель, атака: `type: physical, formula: 5 + 0.5*battlePower, target: enemy/nearest`
- Цілитель, зцілення: `type: holy, formula: 10 + 1*crystalRegenSpeed, target: ally/any`
- Некромант, самон зомбі: `effect_kind: summon, target: self/corpse_of_dead_ally, formula: zombie_stats = fn(lord)`
- Мечник, удар списом по фронту: `range_type: melee, target_rule: {selection:"nearest", count:1, width:2}` — б'є ціль + сусідню клітинку того ж ряду одночасно (не aoe навколо себе, а "широкий" удар по фронту).
- Маг, вогняна куля: `range_type: ranged, target_rule: {selection:"lowest_hp", count:1, include_target_neighbors:true}` — шкода лягає на обрану ціль і на клітинки, сусідні саме до неї, куди б її не поставили.
- Страж, `positional_modifiers: []`, але `passives: [Guardian{pct:0.5}]` — 50% шкоди, спрямованої на сусідні клітинки, перехоплює на себе.

### Примітки щодо фракцій
- Лорд не має єдиного `faction` — натомість `primary_faction` (дає доступ до бойового бонусу, п.1) і `secondary_factions` (лише розширює список доступних для найму фракцій, без бонусу). Найм фракцій поза цим списком — заборонений або можливий через спецнавик лорда/будівлю (гачок для розширення, не хардкод).
- `race_type` — окремий вимір від `faction`: дозволяє синергії/навички ловити ціль за расою незалежно від того, хто її найняв (напр. "призов зомбі можливий з трупу будь-якого race_type: human або beast, але не construct").

### Примітки щодо балансу
- `resistances` — фіксовані модифікатори шкоди (наприклад флет-редукція чи %), не масштабуються характеристиками лорда — це власна "стеля" юніта.
- `exp_reward_on_kill` окремий від `exp_to_upgrade`: перший — "вартість" юніта як цілі, другий — його "поріг росту".
- Всі формули — просто ще один Modifier у системі з п.2 (priority=0, база), тож навичка лорда/бафф може перекрити конкретну формулу конкретної дії без зміни коду.
